# Schema Guide

This guide explains how to use Archflow schema files.

The goal is to help users understand:

- which files Archflow expects
- which files are required
- in what order they should be written
- what the smallest useful setup looks like

This guide is intentionally practical.
It is not a formal specification reference.
For field-level definitions, see the files under `schemas/`.

---

## Overview

Archflow uses structured definition files to describe architecture before implementation begins.

These files define:

- the project context
- where artifacts should live
- what artifacts should exist
- what responsibilities and constraints they should carry
- how those constraints can be handed to AI systems

The schema files in `schemas/` describe the structure of those definition files.

---

## Schema files

Archflow currently defines the following schema drafts:

- `schemas/project.schema.yaml`
- `schemas/placement-rules.schema.yaml`
- `schemas/contracts-template.schema.yaml`
- `schemas/artifacts-plan.schema.yaml`
- `schemas/contract.schema.yaml`
- `schemas/prompt.schema.yaml`

Not all of these need to be written by a user directly.

Some are primary input files.
Others are generated outputs or supporting structures.

---

## What users usually write

In a normal workflow, users usually write these files first:

1. `project.arch.yaml`
2. `placement.rules.yaml`
3. `artifacts.plan.yaml`

Optionally, users may also write:

4. `contracts.template.yaml`

These files are the main inputs for Archflow planning and scaffold generation.

---

## What Archflow may generate

Archflow is expected to generate or derive files such as:

- artifact contract files
- prompt files
- scaffolded placeholder files
- resolved paths

In other words:

- `contract.schema.yaml` usually describes generated artifact contracts
- `prompt.schema.yaml` usually describes generated AI handoff prompts

They are still important to understand, even if users do not author them by hand every time.

---

## Recommended authoring order

If you are starting from scratch, write files in this order.

### 1. Project definition

Start with the project definition.

File:
- `project.arch.yaml`

This defines the architectural frame of the whole project.

It answers:
- what kind of project this is
- which architecture style it follows
- which modules exist
- what language orientation is assumed

Without this file, the rest of the structure lacks context.

---

### 2. Placement rules

Next, define placement rules.

File:
- `placement.rules.yaml`

This maps roles to locations.

It answers:
- where `entity` artifacts should live
- where `usecase` artifacts should live
- where `controller` artifacts should live

Without placement rules, Archflow cannot consistently resolve paths.

---

### 3. Artifact plan

Then define the planned artifacts.

File:
- `artifacts.plan.yaml`

This lists the implementation units Archflow should scaffold or prepare.

It answers:
- what artifacts should exist
- which module each artifact belongs to
- which role each artifact has
- optional inputs and outputs

Without an artifact plan, Archflow does not know what to create.

---

### 4. Contract templates (optional but recommended)

Then define contract templates if you want role-based defaults.

File:
- `contracts.template.yaml`

This provides reusable defaults by role.

It answers:
- what `usecase` artifacts usually do
- what `entity` artifacts must not do
- what default dependency boundaries should apply

This file is optional, but it is strongly recommended because it makes contracts more consistent.

---

## Minimum useful setup

The smallest useful Archflow setup is:

- `project.arch.yaml`
- `placement.rules.yaml`
- `artifacts.plan.yaml`

This is enough to express:

- project context
- role-to-path mapping
- artifact inventory

A minimal setup may skip `contracts.template.yaml`,
but that usually means contracts must be written more manually later.

---

## Recommended setup

A more complete and recommended setup is:

- `project.arch.yaml`
- `placement.rules.yaml`
- `artifacts.plan.yaml`
- `contracts.template.yaml`

This gives Archflow enough information to support:

- scaffold planning
- role-based contract generation
- prompt generation
- future verification

This is the recommended starting point for most projects.

---

## File roles at a glance

| File | Usually written by user | Usually generated by Archflow | Purpose |
|---|---|---|---|
| `project.arch.yaml` | yes | no | define project context |
| `placement.rules.yaml` | yes | no | map roles to paths |
| `artifacts.plan.yaml` | yes | no | define which artifacts should exist |
| `contracts.template.yaml` | often | no | define default role-based contract behavior |
| `*.contract.yaml` | sometimes | often | define artifact-specific responsibilities and constraints |
| `*.prompt.md` or prompt data | rarely | often | define AI handoff instructions |

## Minimal example

Here is the smallest practical example set.

### `project.arch.yaml`

```yaml
project:
  name: minimal-app
  architecture_style: simple
  language: generic

modules:
  - name: user
    features:
      - create_user
      - user_entity
```

### `placement.rules.yaml`

```yaml
roles:
  usecase:
    path: src/application/usecases/
  entity:
    path: src/domain/entities/
```

### `artifacts.plan.yaml`

```yaml
artifacts:
  - name: create_user
    module: user
    role: usecase
    inputs:
      - CreateUserCommand
    outputs:
      - CreateUserResult

  - name: user
    module: user
    role: entity
    outputs:
      - User
```

---

This is enough to start planning or scaffolding.

## When to add contract templates

You should add `contracts.template.yaml` when:

- you want artifacts of the same role to share defaults
- you want clearer dependency boundaries
- you want prompt generation to be more consistent
- you want future verification to rely on stronger defaults

A contract template becomes especially useful once the same roles appear repeatedly across modules.

---

## Relationship between files

The files work together in this order:

- `project.arch.yaml` defines the context
- `placement.rules.yaml` defines where roles should live
- `artifacts.plan.yaml` defines what should exist
- `contracts.template.yaml` defines default contract behavior
- Archflow generates or refines artifact contracts
- Archflow generates prompts from contracts

This means:

- the project gives meaning
- placement rules give location
- artifacts give execution units
- contracts give boundaries
- prompts give AI handoff

## Which files are required

At the current draft level, the practical required files are:

### Required for a meaningful setup

- `project.arch.yaml`
- `placement.rules.yaml`
- `artifacts.plan.yaml`

### Recommended

- `contracts.template.yaml`

### Usually generated later

- `*.contract.yaml`
- `prompt files`

## Common mistakes

### Writing artifacts before defining roles

If artifact roles are not aligned with placement rules, paths become unclear.

### Treating the project file as a full design doc

The project file should define architectural frame, not full business logic.

### Skipping contracts entirely

Even if templates are optional, the overall Archflow model becomes much weaker without contract definitions.

### Making prompts the source of truth

Prompts should be derived from contracts, not replace them.

### Mixing role naming between files

If one file says usecase and another says application_service for the same concept, consistency becomes difficult.

## Practical starting path

If you want the simplest possible path, use this workflow:

1. create `project.arch.yaml`
2. create `placement.rules.yaml`
3. create `artifacts.plan.yaml`
4. optionally create `contracts.template.yaml`
5. use examples under `examples/`
6. generate or draft contracts
7. generate prompts
8. verify consistency later

This path is enough to begin using Archflow conceptually even before a full CLI exists.

---

## Suggested naming conventions

Use these naming patterns consistently where possible:

- `project.arch.yaml`
- `placement.rules.yaml`
- `artifacts.plan.yaml`
- `contracts.template.yaml`

For generated files:

- `<artifact>.contract.yaml`
- `<artifact>.prompt.md`

These names are readable, stable, and easy to teach.
